1. 테스트 방법론 · 환경

브로커(KRaft 단일 노드, combined broker+controller)를 대상 인스턴스에 배포하고, 별도 benchmark-client 노드(c6in.2xlarge, 같은 AZ)에서 kafka-producer-perf-test → kafka-consumer-perf-test를 5회 반복 실행. 레코드 500만건 × 1KB(≈4.77GiB/회), acks=all linger.ms=10 batch.size=131072. 주 지표: produce MB/s(높을수록 좋음), 보조: produce p99 지연(ms, 낮을수록 좋음), consume MB/s. value = produce MB/s ÷ $.

Phase 2 — 포화(saturation) 시나리오 (§9)

베이스라인은 싱글 producer/consumer + 무압축이라 CPU-bound가 아니라 단일 커넥션의 네트워크 RTT/배치 처리량에 상한이 걸려, 대상 인스턴스의 CPU를 다 못 쓰고 인스턴스 간 차이가 좁게 나온다 (진단 완료). 이 한계를 제거하기 위해 producer/consumer 8-way 병렬 + 토픽 레벨 압축(uncompressed/lz4/zstd, 3회 중앙값)을 추가로 측정했다. 압축은 producer-props가 아니라 토픽 설정으로 강제해, 압축 CPU 비용이 클라이언트가 아닌 측정 대상 브로커(대상 인스턴스)에 실리도록 했다 — 클라이언트는 전 인스턴스 공통(c6in.2xlarge)이라 클라이언트측 압축은 측정 목적 자체를 깨뜨린다.

테스트 도구

Apache Kafka 3.9.1 (producer/consumer-perf-test)

클러스터

EKS / Karpenter

브로커 스토리지

gp3 16000 IOPS / 1000 MB/s (전 인스턴스 동일)

힙 정책

브로커 힙 = RAM의 25% · 나머지는 OS page cache

인스턴스 크기

xlarge (4 vCPU) · C 8GB / M 16GB / R 32GB

워크로드

500만 레코드 × 1KB (~4.77 GiB/회) × 5회

클라이언트

c6in.2xlarge 고정, 브로커와 같은 AZ

리전

ap-northeast-2 (서울)

메모리/힙 설정 (공정성 핵심)

  • Pod 고정 메모리 limit 없음 — 브로커가 노드 전체 RAM을 인식해 나머지(75%)를 OS page cache로 활용. Kafka는 힙보다 page cache 적중률이 처리량을 좌우하므로, C(8GB)/M(16GB)/R(32GB) 차이가 의도된 비교 변수로 작동.
  • 힙 25%(RAM-relative) — 절대 바이트가 아닌 비율이라 인스턴스 크기가 달라도 동일한 정책이 적용됨.
⚠️ 네트워크 교란 — 해석 주의
  • produce/consume 각 ~4.8GB가 네트워크를 지나므로 인스턴스별 네트워크 baseline/burst 대역폭이 결과에 반영됨(구세대 xlarge는 버스트 크레딧 소진 가능). 이는 실제 Kafka 배포에서도 나타나는 특성이라 iperf3 리포트와 교차 참조를 권장.
  • 디스크는 gp3 절대 최대(2000MB/s, 16000 IOPS)로 통일 — 1차 시도(1000MB/s)에서 gen6~8 다수가 그 캡 근처에 몰려 세대 순위가 역전되는 걸 발견해 상향(io2는 계정 단위 IOPS 쿼터 (리전 전체 100,000)에 걸려 54개 병렬 실행 불가로 gp3를 유지). §9 재측정 결과 전 인스턴스가 2000MB/s 캡의 90%에도 못 미쳐(최고 c6in.xlarge 1123MB/s) 스토리지가 더 이상 병목이 아님을 확인 — 지금 보이는 세대별 차이는 순수 하드웨어 특성.
  • ⚠️ burst vs sustained 미확인: AWS 공식 performance-testing-framework-for-apache-kafka는 테스트 전 EC2/EBS 버스트 크레딧을 고갈시켜 sustained 성능을 측정하고 1시간 단위로 테스트함. 우리는 run당 수분짜리라 burst 성능을 측정했을 가능성 있음 — baseline이 낮은 인스턴스(c5.xlarge 등)는 장시간 운영 시 더 낮아질 수 있음.

10. 램프업 · 포화점 · 지연곡선 (Phase 3)

§9 포화 시나리오는 항상 최대치로 밀어붙이는 방식이라 "얼마나 세게 밀면 이 정도 나온다"만 보여준다. 여기서는 AWS 공식 performance-testing-framework-for-apache-kafka의 방법론(점진 램프업 + 정지조건 + 버스트크레딧 고갈)을 축소 적용해 실제/목표 처리량 비율이 99.5% 밑으로 떨어지는 진짜 포화점과 처리량이 오를수록 지연이 어떻게 악화되는지를 본다. 90초 무제한 부하로 버스트 크레딧을 먼저 소진한 뒤, §9 무압축 실측치의 20~160%를 8단계로 증가시키며 측정(1회 측정 — 곡선 자체가 하나의 데이터셋).
